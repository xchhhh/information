# 混合检索（Hybrid Retrieval）设计详解

检索质量是 RAG 答案质量的天花板。本项目采用"稠密向量 + 稀疏关键词"的混合检索，再经 RRF 融合与重排序，最后取 Top-K 拼接上下文。下面逐个说明。

## 一、稠密向量检索（Dense Retrieval）

用 Embedding 模型（部署侧为火山方舟 Doubao Embedding，维度 1024）把问题和资料段落都编码成向量，在 Milvus-Lite 里做最近邻搜索（度量方式 L2）。向量检索擅长捕捉语义相似："大模型应用"和"LLM 开发"语义相近也能召回。

## 二、稀疏关键词检索（BM25）

BM25 是经典的关键词匹配算法，对专名、缩写、编号、技术术语极敏感——比如问题里的 "FastAPI"、"Milvus-Lite"、"systemd" 这类词，向量检索可能漏，BM25 却能精准命中。入库时把"原文 + 来源 + chunk_id"落盘成 bm25_corpus.pkl，检索阶段直接加载做倒排匹配。

## 三、为什么必须混合

单一向量检索会"语义对但词不对"漏掉关键术语；单一 BM25 又无法理解同义改写。两者互补，是工业界 RAG 的标配。

## 四、RRF 倒数排名融合

两条路各自返回带分数的候选列表，但分数量纲不同（向量是距离、BM25 是 TF 统计），不能直接相加。RRF 只取排名：

score(d) = Σ 1 / (k + rank(d))

其中 k=60 是标准常数。排名越靠前贡献越大，且对异常分数鲁棒。融合后得到一个统一的候选列表。

## 五、重排序（Rerank）

融合后的候选再交给跨编码器 BAAI/bge-reranker-v2-m3 做精排。重排模型会把"问题-段落"成对编码，直接输出相关性分数，比双塔向量更准，能把真正相关的段落顶到前面。

## 六、Top-K 截断与窗口大小

- top_k_retrieve=20：每条检索路先各召回前 20，给融合和重排留足候选。
- top_k_final=8：融合 + 重排后，真正送进 LLM 的段落数。

把 top_k_final 从 4 提升到 8，是本项目一次关键优化：在小语料下 top_k=4 会漏掉约 1/3 的相关段落，扩到 8 后相关段落覆盖率从 66.7% 提升到 100%。代价是上下文变长、可能引入少量噪声，但实测检索耗时几乎不变（已缓存首次检索），收益明显大于成本。

## 七、可复现的检索质量度量

为量化检索效果，本项目用确定性指标"相关段落覆盖率"：对每个问题，用稠密向量在 Milvus 中检索出最相似的 10 个段落作为标准答案集，再看混合检索 top-k 命中其中几个。由于混合检索返回排序列表、top-8 是 top-4 的超集，覆盖率@8 ≥ 覆盖率@4 在数学上保证成立，结果完全可复现，不依赖大模型裁判打分。
