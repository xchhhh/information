# 安全保障：鉴权、限流与上传防护

个人服务暴露在公网，安全不能裸奔。本项目在接口层、限流层、上传层都做了防护。

## 一、X-API-Key 接口鉴权

所有写操作与核心接口（/chat、/admin/*）都要求请求头携带合法的 X-API-Key。后端通过依赖项 get_api_key 统一校验：

- 请求必须带 X-API-Key 头，且值命中配置的密钥列表（api_keys）；
- 缺失或错误直接返回 401/403，拒绝处理；
- 聊天页与后台页的"页面"本身是静态文件、无需鉴权，但调用接口必须鉴权，做到"看页面免费、调接口要钥匙"。

密钥写在 .env 的 api_keys 里，部署前务必改成强随机值，不要用默认的 change_me。

## 二、按 IP 限流

为防止单个来源刷接口（尤其是会触发 LLM 计费的 /chat），后端按客户端 IP 做限流：

- 用内存限流器（api/rate_limit.py 的 limiter）记录每个 IP 的请求频率；
- 超过阈值返回 429 Too Many Requests；
- 限流在 /chat 入口最先执行，先于真正的问答逻辑，从源头挡掉恶意流量。

## 三、上传安全（后台 /admin/upload）

后台允许上传资料入库，是最高危的入口，做了三重防护：

1. 类型白名单：只接受 .txt / .md / .pdf，其他扩展名直接 400 拒绝，避免上传可执行文件或脚本。
2. 大小上限：单个文件 20MB，超限返回 413，防止超大文件打爆内存。
3. 路径穿越防护：取文件名时只保留最后一段（os.path.basename），并剥离 ".." 与路径分隔符，把空格转下划线，确保文件只会落到 data/raw 目录、绝不会被写到任意系统路径（防 ../ 穿越攻击）。
4. 多文件支持：一次可上传多个文件，逐个校验、逐个保存，返回成功列表。

## 四、入库重建保护

/admin/ingest 带 reset 参数：reset=true 时先 drop_collection 清空旧集合再重新入库，避免向量重复导致检索质量下降。该接口同样受 X-API-Key 保护，不会被人随意触发重建。

## 五、密钥不进仓库

.env 已被 .gitignore 忽略，密钥不会随代码推送到 GitHub / Gitee。配置与密钥分离，是基本的安全卫生习惯。
