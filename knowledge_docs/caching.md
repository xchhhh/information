# 语义缓存（Semantic Cache）

为降低重复问题的计算与调用成本，本项目实现了语义缓存（Semantic Cache）模块。

## 一、为什么需要缓存

RAG 每次问答都要走：Embedding（把问题向量化）+ 混合检索（Milvus + BM25）+ 重排 + LLM 生成。其中 Embedding 与 LLM 调用都有成本（时间 + 费用）。而真实场景下，大量问题是"高度相似"的：

- "你用什么向量库？" 和 "向量数据库用的什么？" 语义几乎一样；
- 考官连续追问时，问题常是同一主题的变体。

如果每次都全链路重算，是浪费。

## 二、工作原理

语义缓存不比较"字面是否相同"，而是比较"语义是否相近"：

1. 用户提问进来，先对问题做 Embedding 得到问题向量；
2. 在缓存里用余弦相似度比对历史问题的向量；
3. 若最高相似度 ≥ 阈值（本项目设为 0.95），直接返回该历史问题对应的答案，跳过检索与生成；
4. 若未命中，则走完整 RAG 链路，并把"问题向量 + 答案 + 来源"写入缓存，供下次复用。

## 三、阈值的选择

- 阈值越高（越接近 1），只在"几乎完全同义"时才复用，准确率最高但命中率低；
- 阈值越低，命中率上升但可能把"不同问题"误判为相同，返回错误答案；
- 本项目取 0.95，偏向"宁可少命中、不可错复用"，保证答案正确性优先。

## 四、收益与状态

- 收益：对相似问题复用历史答案，显著降低重复的 Embedding 与 LLM 调用，提升响应速度、节省费用。
- 当前状态：缓存模块已实现，阈值 0.95 已配置；开关（cache.enabled）默认关闭，可按需开启。关闭是为了在评估 / 调试阶段保证每次都走真实检索链路、结果稳定可复现；上线后开启即可享受缓存收益。

## 五、与检索质量的配合

缓存位于检索之前，是"快路径"；未命中才进入"慢路径"（混合检索 + 重排 + 生成）。二者不冲突：缓存解决"重复问题"，检索解决"新问题"，共同构成完整问答能力。
