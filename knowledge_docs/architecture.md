# 系统总体架构与端到端链路

本项目是一套面向个人作品集 / 简历展示的 RAG（Retrieval-Augmented Generation，检索增强生成）知识库问答系统。它的核心目标是：把散落在简历、项目文档里的个人资料，变成"能语义检索、能智能问答"的可交互知识库，让考官或访客直接通过对话获取信息。

## 一、为什么要做 RAG 而不是纯微调

个人资料更新频繁、体量小、强调"答案必须忠实于原文"。纯大模型微调成本高、易幻觉、难迭代。RAG 把"检索"和"生成"解耦：先用检索把最相关的资料片段找出来，再让大模型只基于这些片段作答，从根本上抑制编造。

## 二、端到端处理链路

一次用户提问，从进到出经历以下阶段：

1. 接收问题：前端通过 POST /chat 把用户 query 发给后端。
2. 语义缓存命中判断：先用问题的向量去语义缓存（Semantic Cache）里比对，相似度超过阈值（0.95）则直接复用历史答案，跳过后续检索与生成，显著降低 Embedding 与 LLM 调用成本。
3. 查询改写（Query Rewrite）：对原始问题做同义扩展与指代消解，提升后续检索的召回率。例如把"你用什么向量库"改写成更利于向量匹配的表述。
4. 混合检索（Hybrid Retrieval）：同时走两条路——稠密向量检索（Milvus-Lite）与稀疏关键词检索（BM25），各自先召回前 20 个候选。
5. 倒数排名融合（RRF）：用 RRF（Reciprocal Rank Fusion，常数 k=60）把两条路的排序结果融合成一个列表。RRF 不依赖分数绝对值，只依赖排名，因此对异质分数最稳健。
6. 重排序（Rerank）：用跨编码器重排模型 BAAI/bge-reranker-v2-m3 对融合后的候选做二次精排，按"段落与问题的真实相关性"重排。
7. 上下文截断：融合 + 重排后，取相关性最高的前 8 个段落（top_k_final=8）拼成上下文窗口。
8. 上下文增强与生成：把段落编号、来源拼进 Prompt，交给 DeepSeek 生成答案，并严格约束"只依据资料、资料缺失就明说没有"。
9. 返回：答案 + 引用来源（资料编号）回传给前端卡片式对话界面。

## 三、为什么是这条链路

- 混合检索弥补了单一向量检索对专名、编号不敏感的短板；
- RRF 融合避免两条路分数量纲不一致；
- 重排把"看着像"的候选压到后面，让真正相关的段落进入 top-8；
- top_k 从 4 提升到 8，是为了在"上下文更全"和"噪声更少"之间取平衡，实测能显著提高相关段落覆盖率。

整条链路全部通过 FastAPI 暴露为 REST 接口，前端原生 HTML/CSS/JS 同源托管，部署在腾讯云服务器以 systemd 常驻运行。
